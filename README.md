# Agent 邮件域服务项目(AgentMailbox)文档

## 1. 项目背景

随着 AI Agent 在业务场景中的使用越来越多，Agent 不再只是一个被动调用的模型能力，而是开始承担真实的业务角色，例如：

- 销售助理
- 客服助理
- 招聘助理
- 采购助理
- 通知与跟进机器人

在这些场景下，Agent 往往需要具备“像一个真实员工一样通过邮箱沟通”的能力。  
也就是说，Agent 需要有自己的邮箱地址，能够：

- 接收外部邮件
- 读取邮件内容
- 理解上下文
- 回复邮件
- 发起新的邮件沟通
- 保留会话记录，便于持续处理

因此，本项目希望建设一个面向 Agent 的**邮件域服务**，为 Agent 提供统一的邮箱接入和邮件收发能力。

---

## 2. 项目目标

本项目的目标，是搭建一个**轻量、可控、可扩展**的 Agent 邮件域服务，使系统能够支持以下核心能力：

### 2.1 为 Agent 提供专属邮箱
例如：
- `sales-agent@local.ai`
- `support-agent@local.ai`

每个 Agent 都可以拥有自己的邮箱地址，用于对外通信。

### 2.2 支持邮件接收
系统能够接收入站邮件，并识别邮件属于哪个 Agent 邮箱。

### 2.3 支持邮件读取与管理
系统能够保存邮件内容、主题、发件人、时间、附件等信息，供 Agent 后续读取和处理。

### 2.4 支持邮件回复与发送
Agent 不仅可以读取邮件，还可以通过系统回复邮件，或者主动发起新邮件。

### 2.5 支持基本追踪和状态管理
系统需要知道每封邮件当前是否已处理、是否发送成功、是否失败，以便后续重试和管理。

---

## 3. 项目定位

这个项目的定位不是去做一个完整的企业级邮件服务器，而是做一个：

> **面向 AI Agent 使用的邮件基础服务**

它更关注的是：

- Agent 如何方便地读取邮件
- Agent 如何方便地发邮件和回复邮件
- 如何保存上下文和线程关系
- 如何让邮件处理可追踪、可管理

而不是追求传统邮箱系统那种非常完整的终端兼容能力。

简单来说，本项目更像是：

> **Agent 的邮件基础设施**

---

## 4. 项目范围

## 4.1 本项目包含的内容

### 邮件接收
系统能够接收发送到指定域名和邮箱地址的邮件。

### 邮件解析与存储
系统能够保存邮件正文、主题、地址、附件等基础信息。

### Agent 读取接口
系统提供接口，方便 Agent 查询待处理邮件、查看邮件详情、查看会话线程。

### 邮件发送能力
系统提供接口，支持 Agent 发送新邮件和回复邮件。

### 状态管理
系统支持邮件状态流转，例如未处理、处理中、已处理、发送成功、发送失败等。

### 后台任务处理
系统具备基础的后台任务能力，用于发送队列、失败重试等工作。

---

## 4.2 本项目暂不包含的内容

为了控制项目复杂度，当前版本不考虑以下内容：

- 完整的企业邮箱能力
- IMAP / POP3 兼容
- 复杂邮件安全策略
- 多租户复杂隔离
- 杀毒、反垃圾能力
- 公网生产级高可用能力
- 完整 Web 管理后台

这些内容可以在后续版本中按需要逐步扩展。

---

## 5. 核心需求

本项目的核心需求可以概括为以下几个方面。

## 5.1 邮箱管理
系统需要支持创建和管理 Agent 对应的邮箱，例如一个 Agent 绑定一个邮箱地址。

## 5.2 邮件接入
外部发来的邮件能够被系统接收，并路由到正确的 Agent 邮箱。

## 5.3 邮件存储
接收到的邮件需要被完整保存，至少包括：

- 发件人
- 收件人
- 主题
- 正文
- 时间
- 附件
- 原始邮件内容

## 5.4 邮件线程关系
系统需要能够识别回复链路，把同一轮沟通归到同一个会话线程下，方便 Agent 理解上下文。

## 5.5 Agent 可操作性
Agent 能够通过 API：

- 查看待处理邮件
- 查看邮件详情
- 查看某个线程的历史记录
- 标记处理状态
- 回复邮件
- 发起新邮件

## 5.6 出站可靠性
Agent 发出的邮件不能只是“发起请求”，系统还要负责真正发送，并在失败时支持重试。

## 5.7 可追踪性
邮件从接收到处理、再到回复，都需要有基本的状态和日志，便于排查问题和管理。

---

## 6. 建设原则

为了确保项目能够快速落地，同时具备足够的可维护性，本项目遵循以下原则：

### 6.1 简单优先
优先实现 Agent 真正需要的核心能力，不追求一开始做成复杂的大系统。

### 6.2 可用优先
先保证“能收、能读、能发、能回”，再逐步增强能力。

### 6.3 本地化和可控
尽量使用自建、可控的技术方案，避免依赖过多外部组件，便于内部部署和维护。

### 6.4 面向 Agent
接口和数据组织方式优先考虑 Agent 的使用体验，而不是传统邮箱客户端的使用方式。

### 6.5 可扩展
虽然当前版本做轻量方案，但设计上要为后续增加能力保留空间。

## 7. 技术栈

本项目采用纯 .NET 技术路线，保持整体一致性和部署简洁性。

### 主要技术栈如下：

- **ASP.NET Core**
  - 用于提供系统 API
  - 包括 Agent 使用的接口，以及管理用途的接口

- **BackgroundService**
  - 用于处理后台任务
  - 如邮件发送队列、失败重试等

- **TcpListener**
  - 用于实现基础邮件接收服务

- **MimeKit**
  - 用于解析接收到的邮件内容

- **MailKit**
  - 用于发送邮件

- **Dapper + SQLite**
  - 用于保存邮箱、邮件、线程、队列等基础数据

- **本地磁盘存储**
  - 用于保存原始邮件和附件

- **ILogger**
  - 用于记录运行日志

这套技术方案的优点是：

- 技术栈统一
- 学习和部署成本低
- 适合单机和轻量级场景
- 非常适合作为 MVP 或内部系统起步方案

## 8. 总体方案

本项目整体上可以理解为一个“邮件接收 + 邮件存储 + Agent 接口 + 邮件发送”的闭环服务。

## 8.1 入站邮件流程
基本流程如下：

1. 外部邮件发送到系统域名下的某个邮箱地址
2. 系统接收邮件
3. 系统识别目标邮箱属于哪个 Agent
4. 系统解析邮件内容并保存
5. 邮件进入待处理状态
6. Agent 通过接口读取邮件并进行处理

## 8.2 出站邮件流程
基本流程如下：

1. Agent 发起发送或回复请求
2. 系统生成待发送邮件
3. 邮件进入发送队列
4. 后台任务执行实际发送
5. 成功则更新状态
6. 失败则记录原因，并根据规则重试

## 8.3 线程与上下文
为了让 Agent 能理解同一轮沟通的上下文，系统会将有回复关系的邮件归到同一个会话线程中。  
这样 Agent 在处理某封邮件时，不仅能看到当前邮件，还能查看整个沟通过程。

## 10. 数据与状态的基本思路

虽然当前版本不展开详细数据结构，但系统会围绕几个核心对象组织信息：

- **邮箱**：对应某个 Agent 的邮箱地址
- **邮件**：每一封收到或发出的邮件
- **线程**：同一轮沟通的邮件集合
- **附件**：邮件附带的文件
- **发送任务**：待发送或失败重试的任务

同时，每封邮件都会有基本状态，例如：

### 入站邮件状态
- 未处理
- 处理中
- 已处理
- 失败
- 忽略

### 出站邮件状态
- 待发送
- 发送中
- 已发送
- 发送失败

这样的状态设计是为了保证邮件处理过程可见、可管、可恢复。

## 11. 为什么采用这套方案

选择这套方案，主要基于以下考虑：

### 11.1 满足当前核心诉求
当前最重要的是让 Agent 拥有邮箱能力，而不是一开始就做非常复杂的邮件平台。

### 11.2 降低实施复杂度
采用 .NET + 本地数据库 + 本地文件的方案，开发和部署都比较直接，适合快速落地。

### 11.3 方便内部环境使用
如果系统主要部署在内网、测试环境或受控环境，这种轻量化方案非常合适。

### 11.4 便于后续演进
未来如果规模扩大，也可以逐步升级：
- 数据库升级
- 存储升级
- 增加安全能力
- 增加管理界面
- 增加更复杂的路由与自动化处理

## 13. 当前版本的边界

为了保证项目可控，当前版本建议聚焦在“核心闭环跑通”，即：

- 能创建邮箱
- 能接收邮件
- 能让 Agent 读取邮件
- 能让 Agent 回复和发信
- 能保存记录
- 能处理基础失败重试

而不建议在第一阶段加入过多扩展目标，例如：

- 企业级安全网关
- 公网邮件信誉体系
- 完整邮箱客户端兼容
- 大规模分布式部署

这些都可以作为后续阶段的增强方向。

## 15. 结论

本项目本质上是在建设一套：

> **让 AI Agent 具备真实邮件交互能力的基础服务**

它的价值在于：

- 让 Agent 真正能参与真实世界的邮件沟通
- 把邮件收发、上下文保存、状态追踪统一起来
- 为后续更复杂的 Agent 自动化工作流打下基础

从方案上看，采用纯 .NET 的轻量技术栈，能够很好地平衡：

- 实施成本
- 可控性
- 可维护性
- 后续扩展空间

因此，这个项目适合作为 Agent 邮件能力建设的第一阶段方案。
